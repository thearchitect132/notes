Potential Futures, Latent Actions, and Identity‑Adjacent Possibility Space



1\. Purpose of Figments

A Figment is the smallest representable unit of potential action in the Civilisation‑Stack.



Where:



the Identity Vector represents who you are



the Trait Vector represents how you behave



the Arena represents where you act



…the Figment represents:



What you could do next.



A Figment Constellation is the structured set of all figments associated with an identity — its adjacent possible.



Figments allow the civilisation to:



model unrealised futures



evaluate possible actions



simulate branching trajectories



reason about latent capability



maintain continuity across transformation



They are the possibility substrate of GLIRL.



2\. What a Figment Is

A Figment is a structured GLIRL object:



Code

FIGMENT(intent, capability, context, narrative\_position)

It encodes four dimensions:



Intent — what the identity wants to do



Capability — what the identity can do



Context — where the action would occur



Narrative Position — how the action fits the story



Each dimension is itself a GLIRL‑evaluated structure.



3\. Figments vs Actions

A Figment is not an action.



An action is real, witnessed, and state‑changing



A figment is hypothetical, uncommitted, and reversible



Actions update the Identity Vector.

Figments update the Figment Constellation.



This separation allows the civilisation to:



explore futures without committing



evaluate consequences before acting



maintain narrative coherence



avoid irreversible mistakes



Figments are the safe sandbox of identity evolution.



4\. Figment Constellation Structure

A Figment Constellation is a graph, not a list.



Nodes:



individual figments



Edges:



structural relationships between figments



Edges encode:



compatibility



mutual exclusion



narrative adjacency



capability dependency



arena relevance



The constellation is the map of possible futures.



5\. GLIRL Operators for Figments

Figments use a dedicated operator family:



5.1 POSSIBLE

Code

POSSIBLE(figment) := ELIGIBLE(GLIRL\_conditions)

Determines whether a figment is structurally allowed.



5.2 COMPATIBLE

Code

COMPATIBLE(f1, f2) := ALIGN(f1, f2)

Two figments can co‑exist in the constellation.



5.3 EXCLUDES

Code

EXCLUDES(f1, f2) := AND(f1, NOT(f2))

Choosing one eliminates the other.



5.4 ADVANCES

Code

ADVANCES(f1 → f2) := CONTINUES(f1 → f2)

A figment leads naturally to another.



5.5 BREAKS

Code

BREAKS(f1 → f2) := BREAKS\_ARC(f1, f2)

A figment disrupts the narrative trajectory.



These operators allow GLIRL to reason about potential futures with the same rigour it applies to actual state transitions.



6\. Figment Lifecycle

A figment moves through four phases:



6.1 Emergence

A new figment appears when:



intent changes



capability expands



context shifts



narrative position evolves



6.2 Evaluation

GLIRL evaluates:



invariants



compatibility



narrative coherence



cross‑layer alignment



6.3 Activation

A figment becomes an action candidate when:



Code

ELIGIBLE(POSSIBLE(figment)) = TRUE

6.4 Collapse

A figment collapses when:



it is chosen (becomes an action)



it becomes incompatible



it violates invariants



narrative structure shifts



The constellation is always in motion.
